# Defect Lifecycle Intelligence Agent

## Project Overview

A multi-agent system for JIRA defect lifecycle intelligence, built on the **Knowledge Governance & Organizational Learning** architecture (Version 4.0).

### Evolution Roadmap

```
v1.0 Workflow Intelligence
    ↓
v2.0 Trustworthy Intelligence
    ↓
v3.0 Case Intelligence
    ↓
v4.0 Knowledge Governance ← Current
    ↓
v5.0 Engineering Digital Twin
```

## Core Design Principles

### Principle 1 — Fact Before Interpretation
Facts must always be extracted before compliance evaluation and recommendation generation.
- No Agent may modify or reinterpret Facts generated by Fact Agent.

### Principle 2 — Rules Before Reasoning
Compliance decisions are produced by Rule Engine.
- LLMs may explain decisions but may never generate compliance decisions independently.

### Principle 3 — Human Authority Supremacy
Human reviewers remain the final authority.
- All AI recommendations are advisory.

### Principle 4 — Knowledge Must Evolve
Historical decisions, architectural patterns, and governance knowledge must continuously improve the platform.

### Principle 5 — Ontology Is the Single Source of Truth
All reusable knowledge must be represented using canonical ontology entities.
- Free-text knowledge is considered transient and non-governable.

## Agent Specifications

### 3.1 Fact Agent

**Responsibilities:**
- Timeline reconstruction
- Clone topology reconstruction
- Evidence extraction
- TCI calculation
- PFI calculation
- Historical pattern retrieval

**Output:** Fact JSON

**Prohibited Actions:**
- Risk grading
- Limitation eligibility decisions
- Compliance decisions

### 3.2 Review Agent

**Responsibilities:**
- Compliance verification
- Limitation eligibility evaluation
- Security policy enforcement
- Evidence completeness validation

**Supported Decisions:**
- MUST_FIX_BLOCKER
- TEMP_LIMITATION_ELIGIBLE
- CRITICAL_SSRB_REVIEW
- INVALID_CLOSURE_REQUEST
- INSUFFICIENT_EVIDENCE

**Note:** Review Agent operates exclusively through Rule Engine execution.

### 3.3 Advisor Agent

**Responsibilities:**
- Action generation
- Risk communication
- Executive summaries
- TIP drafting

**Inputs:**
- Fact JSON
- Review JSON

**Output:** Three-track recommendation matrix:
- Preferred Path
- Alternative Path
- Escalation Path

**Note:** Advisor Agent may not alter Review Agent decisions.

## Case Intelligence Layer

### Purpose
- Retrieve historical engineering experience
- Convert historical defect resolution patterns into actionable context

### Retrieval Strategy
**Primary Match Dimensions:**
- Technical Domain
- Affected Component
- Failure Signature
- Root Cause Category
- Platform Family

### Similarity Confidence
| Category | Score Range |
|----------|-------------|
| Strong Match | >90% |
| Related Match | 70–90% |
| Reference Match | 50–70% |
| Discarded | <50% |

## Knowledge Governance Layer

### Purpose
Governance as a first-class system component.

### Ontology Shadow Suggestion
When canonicalization fails, Agent may create:
```
Draft_Ontology_Node
├── Name: Draft_Entity_Name
├── Status: PENDING_REVIEW
├── Confidence: 0.87
└── Source Cases: 23
```

### Human Promotion Workflow
```
Draft Node → Architecture Review Board → Approve/Reject → Ontology Update
```
- Only humans may promote ontology nodes
- Agents may suggest
- Agents may never publish ontology changes

## Human Feedback Flywheel

### Override Event
When human reviewers override AI recommendations, record:
```json
{
  "systemDecision": "TEMP_LIMITATION",
  "humanDecision": "MUST_FIX",
  "reason": "Customer-visible availability impact",
  "reviewer": "Senior PE",
  "timestamp": "2026-06-15"
}
```

### Correction Pattern Generation
Override events transform into correction patterns:
```json
{
  "pattern": "Customer-visible KVM latency",
  "historicalOutcome": "82% escalated to Must Fix",
  "confidence": 0.91
}
```

## Architecture Inheritance Engine

### Purpose
Handle new platforms with no historical cases.

### Inheritance Graph Example
```
Platform_GNR → Platform_BHS → Platform_Next
                    ↑
            Inheritance Weight: 75%
```

### Ancestor Retrieval
When no local cases exist, retrieve:
- Parent Platform Cases
- Shared Component Cases
- Shared Technical Domain Cases

### Generalized Knowledge Transfer
Historical defects may be reused when:
- Shared Component Similarity > 70%
- OR Architecture Inheritance Weight > 60%

## Confidence Escalation Framework

### Confidence Categories
| Category | Range |
|----------|-------|
| High Confidence | 0.90–1.00 |
| Medium Confidence | 0.75–0.89 |
| Low Confidence | <0.75 |

### Escalation Rules
- Low confidence canonicalization → Human Review
- Low confidence ontology match → Draft Ontology Queue
- Low confidence recommendation → Advisory Warning

## Knowledge Store Architecture

### Supported Objects (Structured Only)

- Ontology Entities
- Correction Patterns
- Architecture Graphs
- Historical Cases
- Governance Decisions

**Note:** Raw Jira comments are NOT stored as knowledge.
